# 横切：并发与冲突处理

> 多管理员协作、AI Job 抢占、考试限时锁、SRS 更新、流水写入的并发策略。
> 通用原则：**学员侧写流水（append-only）+ 行级乐观锁；管理端写内容侧"最后保存为准 + 提示"+ 派生版本**。

---

## 1. 管理端编辑冲突（KP / Question / Lesson / Exam / Prompt）

### 策略：乐观锁 + 最后保存为准 + 提示

| 步骤 | 行为 |
|------|------|
| 1. 读 | 客户端拿到 `updated_at` 时间戳 |
| 2. 写 | 请求 header 必须带 `If-Match: <ISO-8601 updated_at>` |
| 3. 服务端 | `UPDATE ... WHERE id=? AND updated_at=?` ；影响行数 = 0 → `409 COURSE_STALE_VERSION` |
| 4. 客户端 | 提示"另一管理员已更新，可选 [覆盖] 或 [刷新查看]" |
| 5. 覆盖 | 不带 `If-Match` 重发，服务端跳过校验直接覆盖 |

**Hono middleware 伪码**：

```ts
if (req.method === 'PATCH' && req.headers['if-match']) {
  const result = await db.update(...)
    .set(...).where(eq(table.id, id), eq(table.updated_at, ifMatch));
  if (result.rowCount === 0) throw new ConflictError('COURSE_STALE_VERSION');
}
```

### 派生版本（内容字段）

详见 [05 §A8 KP 编辑](./05-管理端-KP与题目CRUD.md)。
关键：`reviewing/approved` 状态下编辑 → 生成新行 + `version+=1`，不破坏审核流。

## 2. AI Job 抢占（worker 双消费防护）

```sql
-- worker 抢占 1 条 queued
update zhiyu.course_ai_generation_jobs
set status = 'running', started_at = now()
where id = (
  select id from zhiyu.course_ai_generation_jobs
  where status = 'queued'
  order by created_at
  for update skip locked
  limit 1
)
returning *;
```

`FOR UPDATE SKIP LOCKED` 保证多 worker 不抢同一行。

### 心跳超时回收

后端 cron 每分钟扫一次：
```sql
update zhiyu.course_ai_generation_jobs
set status = 'queued', started_at = null
where status = 'running'
  and started_at < now() - interval '10 minutes';
```

> worker 必须 ≤ 5 分钟一次心跳（[I3](./06-管理端-AI工作台.md)）。

### 幂等

`idempotency_key` 唯一索引确保前端重复点击不会创建多个 job。
重试用 `:retry-N` 后缀（[06 §A18.3](./06-管理端-AI工作台.md)）。

## 3. 考试限时锁

### 状态转移行锁

`fn_submit_exam` / `fn_abandon_exam` 内：

```sql
select * from zhiyu.course_user_exam_attempts
where id = $1 and user_id = auth.uid()
for update;

if state <> 'in_progress' then
  raise exception 'COURSE_EXAM_NOT_IN_PROGRESS';
end if;

if now() > expires_at then
  update zhiyu.course_user_exam_attempts
  set state='expired', finished_at=now(), score=...
  where id = $1;
  raise exception 'COURSE_EXAM_EXPIRED';
end if;

-- 正常 submit
```

### 超时 cron 与提交并发

如果 cron 在 worker 提交的同一秒执行：行锁 `for update` 串行化，先到先 win，后到的看到 `state='expired'` 则抛 `410 COURSE_EXAM_EXPIRED`。

## 4. SRS 更新（多端并发答题）

学员同设备/多设备同时答题（罕见但需考虑）：

```sql
-- fn_srs_update RPC 内部
insert into zhiyu.course_user_srs (user_id, kp_id, box, due_at, ...)
values ($1, $2, 1, now() + interval '1 day', ...)
on conflict (user_id, kp_id) do update set
  box = case when excluded.is_correct
             then least(zhiyu.course_user_srs.box + 1, 5)
             else 1 end,
  due_at = ...,
  correct_streak = case when excluded.is_correct
                        then zhiyu.course_user_srs.correct_streak + 1
                        else 0 end,
  wrong_count = zhiyu.course_user_srs.wrong_count + case when not excluded.is_correct then 1 else 0 end,
  last_seen_at = now(),
  last_q_type = excluded.last_q_type;
```

`UPSERT` 在行锁下原子执行，无双计风险。

## 5. 课程目录调序（A3 / A4）

```sql
-- 事务内
begin;
  -- 锁全部受影响行
  select * from zhiyu.course_chapters
  where stage_id = $1
  for update;

  -- 按数组顺序更新 chapter_no + sort_order
  update zhiyu.course_chapters set chapter_no = i, sort_order = i * 10
  from unnest($2::uuid[]) with ordinality as t(id, i)
  where zhiyu.course_chapters.id = t.id and zhiyu.course_chapters.stage_id = $1;
commit;
```

并发调序由"最后提交者赢"，无冲突检测（前端展示当前顺序，每次提交全量数组）。

## 6. 节绑定 KP（A4.2）

事务内 DELETE + INSERT：

```sql
begin;
  delete from zhiyu.course_lesson_kp where lesson_id = $1;
  insert into zhiyu.course_lesson_kp (lesson_id, kp_id, card_order)
  select $1, kp_id, card_order from jsonb_to_recordset($2)
    as t(kp_id uuid, card_order int);
commit;
```

无冲突检测；并发提交"最后赢"。

## 7. Prompt 启用切换（A16）

```sql
begin;
  update zhiyu.course_ai_prompts
  set is_active = false
  where scope = (select scope from zhiyu.course_ai_prompts where id = $1)
    and is_active = true;

  update zhiyu.course_ai_prompts
  set is_active = true
  where id = $1;
commit;
```

唯一索引 `uq_course_prompts_one_active_per_scope` 保底防双 active。

## 8. 流水写入（course_user_answers）

- INSERT-only，无并发冲突
- 月分区表，写入分散到当月分区，无热点
- 单用户高频提交：限流由 API 层 600/分负责

## 9. RLS 与并发的相互作用

- 学员"自读自写"靠 `auth.uid()`，RLS 内嵌于查询计划，零额外锁
- 管理端走 `service_role`，绕过 RLS，依赖应用层鉴权 + 业务校验

## 10. 错误码

| code | HTTP | 描述 |
|------|------|------|
| `COURSE_STALE_VERSION` | 409 | 编辑冲突（If-Match 不一致） |
| `COURSE_EXAM_NOT_IN_PROGRESS` | 409 | 状态机不允许操作 |
| `COURSE_EXAM_EXPIRED` | 410 | 已超时 |
| `COURSE_AI_JOB_DUPLICATE` | 409 | idempotency_key 已存在 |
| `COURSE_PROMPT_ACTIVE_CONFLICT` | 409 | 同 scope 启用切换冲突（实际不应发生） |
