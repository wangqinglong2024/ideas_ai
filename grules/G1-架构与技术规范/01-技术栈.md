## 一、整体原则

1. **前端 TypeScript strict**；**后端 Node.js + JavaScript（ESM）**，以 JSDoc + Zod 替代 TS 做类型守护（用户偏好）。
2. **Docker-only**：所有依赖、构建、测试、运行均在容器内完成，本地不直装 Node/Postgres。
3. **Supabase first**：DB / Auth / Storage / Realtime / pgvector 一律走自托管 Supabase。
4. **缺 Key 不阻塞**：第三方服务统一通过 Adapter 接口注入，缺 Key 自动 fallback 到 mock 实现。
5. **AI 工作流二分**：复杂状态机/多智能体 → LangGraph；单次 prompt/流式 → Vercel AI SDK。

## 三、后端技术栈

| 层级 | 选型 | 版本 | 选择理由 |
|------|------|------|---------|
| 运行时 | Node.js | 20.x LTS（`node:20-alpine` 镜像）| 长期支持 |
| 语言 | JavaScript（ESM, `.mjs` / `"type":"module"`）| ES2023 | 用户偏好 JS |
| 类型守护 | JSDoc + Zod | — | 编辑器智能提示 + 运行时校验，弥补无 TS |
| Web 框架 | Express | 4.x | 生态成熟、中间件丰富、与 Supabase 无侵入 |
| API 模式 | REST（OpenAPI 3.1 描述）| — | 与前端 TanStack Query 简单契合 |
| 入参校验 | Zod | 3.x | 与前端表单共用 schema 字符串源 |
| ORM / 查询 | Drizzle ORM（drizzle-kit 迁移）| latest | 轻量、SQL-near、对 Supabase Postgres 友好 |
| 队列 | BullMQ | 5.x | 复用 Redis；AI 工作流离线任务 |
| 调度 | node-cron | 3.x | 周期任务 |
| 日志 | pino | 9.x | 结构化 JSON、低开销 |
| 实时 | Supabase Realtime | — | 消息推送、IM、协作进度 |
| 文件存储 | Supabase Storage SDK | latest | 头像、课程音频、用户上传 |
| 测试 | Vitest + Supertest | latest | 端到端接口测试，容器内执行 |

> **JS-only 例外**：Drizzle schema 以 `.js` 编写并配合 JSDoc；如某文件强类型收益显著（典型：DB schema、共享 Zod），允许使用 `.ts` 但仅限于 `packages/shared` 共享包，由 PM 单点放行。详见 [99-待确认问题清单.md#Q1](./99-待确认问题清单.md)。

## 四、数据 / 认证 / 存储（统一 Supabase 自托管）

| 用途 | 技术 | 备注 |
|------|------|------|
| 关系型主库 | Supabase Postgres | 16.x；schema 名 `zhiyu_dev`（唯一 dev 环境） |
| 用户认证 | Supabase Auth (GoTrue) | 邮箱 / 手机号 / OAuth |
| 后台认证 | 复用 Supabase Auth + RBAC 表 | `admin` 角色 |
| 文件存储 | Supabase Storage | bucket：`avatars` / `course-media` / `user-upload` / `system` |
| 向量检索 | pgvector | RAG / 语义检索 |
| 全文检索 | Postgres FTS | v1 用 zh/vi/th/id 简化分词 |
| 实时推送 | Supabase Realtime | 通知 / IM |
| 缓存 / 队列 | Redis | 复用 Docker 容器，BullMQ + 短期缓存 |
| Edge Functions | Supabase Edge Functions | 第三方 webhook 转发 |

> **不引入**：Pinecone、Upstash、Meilisearch、Cloudflare KV。

## 五、AI 工作流

| 用途 | 选型 | 备注 |
|------|------|------|
| 复杂工作流编排 | LangGraph (TS Runtime, 通过 `tsx` 子进程或独立 worker) | 多步、状态机、多智能体 |
| 简单 LLM 调用 / 流式 | Vercel AI SDK | 纯 prompt → 单次/流式响应 |
| 向量数据库 | Supabase pgvector | 用户偏好 |
| Adapter 层 | `LLMAdapter` / `WorkflowAdapter` / `EmbeddingAdapter` / `TTSAdapter` / `ASRAdapter` | 缺 Key 自动 fallback 到 fixture |
| 主创作模型 | Anthropic Claude Sonnet（待 Key）| `env.md` 缺 → mock |
| 副创作模型 | DeepSeek V3（待 Key）| 缺 → mock |
| 评估 | LangSmith（可选）| v1 不强制 |

> **禁用**：Dify、NocoBase、OpenClaw（参考 `env.md`，已标注停用）。

## 七、版本锁定与升级策略

1. 全部依赖在 `package.json` 中**精确版本**（不写 `^` / `~`），由 `pnpm-lock.yaml` 锁定。
2. 主版本升级须先在 PR 描述中列出 breaking change，并经过 PM 单点放行。
3. 每月一次安全补丁批量升级，CI 跑全量测试通过后合入。
