## 一、通用规范

1. **格式化**：Prettier 单一来源，`.prettierrc` 在仓库根；CI 强制 `--check`。
2. **Lint**：ESLint flat config + `@typescript-eslint`，全栈共用基础规则集；前端追加 React/JSX 规则。
3. **行长**：120 列。
4. **缩进**：2 空格；JSON/YAML 同。
5. **引号**：JS/TS 单引号，JSX 属性双引号。
6. **导入顺序**：node 内置 → 第三方 → `@/*` 别名 → 相对路径；空行分组。
7. **TODO**：必须带 `// TODO(name, 2026-mm): reason`；CI 扫描禁止超过 30 天的 TODO。
8. **不得使用**：`var`、`==`（用 `===`）、`any`（不可避免时 `unknown` + 即时收敛）、`@ts-ignore`（必须 `@ts-expect-error` + 原因）。
9. **国际化文案**：UI 任何可见文案禁止硬编码，统一走 `t('namespace.key')`。
10. **日志/错误信息**：技术语境用英文，对用户可见的走 i18n。
11. **语言**：全栈 **TypeScript strict**；**禁止 `.js` / `.mjs` / JSDoc** 替代方案。

## 二、前端（TypeScript）

### 2.1 文件命名

| 类别 | 命名 | 示例 |
|------|------|------|
| 普通模块 | kebab-case | `format-date.ts` |
| React 组件 | PascalCase | `CourseCard.tsx` |
| Hook | camelCase + `use*` | `useGameSession.ts` |
| Zustand store | camelCase + `.store.ts` | `wallet.store.ts` |
| Zod schema | `*.schema.ts` | `course.schema.ts` |
| 测试 | `*.test.ts(x)` | `CourseCard.test.tsx` |

### 2.2 渲染与反爬（强制）

- **必须纯 CSR/SPA**；禁止 Next.js / Remix / Vite SSR plugin / `vite-plugin-ssr` 等任何形式的服务端渲染。
- **禁止数据预注入**：
  - `index.html` 中不得出现 `<script id="__INITIAL_STATE__">` 等业务数据脚本；
  - `vite.config.ts` 不允许向 HTML 注入业务变量（仅允许注入构建版本、Sentry DSN 等非业务静态量）；
  - 路由 loader 必须运行时 `fetch`，不可在打包阶段读取 DB / 文件系统业务数据。
- **静态营销文案**可直接写在 i18n 文件中（属于品牌内容，不构成被爬取价值）。
- **首次加载体验**：用骨架屏 / Suspense 替代 SSR，避免空白。

### 2.3 组件规范

- 一个文件一个公开组件，默认导出该组件，命名导出其类型 / 子件。
- Props 显式类型：`type Props = { ... }`；禁止 `React.FC`（不会自动注入 children）。
- 服务端数据通过 `useQuery` 获取，禁止在组件内直接 `fetch`。
- 副作用集中在自定义 hook；视图组件保持纯展示。
- 列表 key 必须为业务 id，禁止使用数组索引。

### 2.4 类型定义

- `interface` 仅用于面向对象式继承；其余一律 `type`。
- 共享类型放 `packages/shared-schemas`；优先**先定义 Zod schema，再 `z.infer` 出类型**。
- DB 行类型来自 `packages/supabase-client/src/types/database.ts`（Supabase CLI 自动生成）。
- 时间字段类型 `string`（ISO-8601），由层边界处理 Date 对象。

### 2.5 样式

- 仅使用 Tailwind 原子类 + `cn()` 合并；
- 复用样式抽取成 `ui-kit` 组件而非自定义 class；
- 颜色/间距/圆角必须使用 Tailwind theme token，禁止任意值（`text-[#abc]`）除非配色尚未沉淀。

### 2.6 路由

- 路由文件即权限边界：`_auth/` 下默认登录守卫，`_admin/` 下默认 admin 守卫。
- Loader 统一返回 `{ data, deferred? }`，错误抛出由根 boundary 统一处理。

## 三、后端（Hono + TypeScript）

### 3.1 文件与模块

- 全部 ESM TypeScript（`"type": "module"`，源码 `.ts`，编译输出 `.js`）。
- 导入路径**必须**带扩展名：`import { x } from './foo.ts'`（Node 20 + `--experimental-strip-types` 或编译后 `.js`，由 `tsconfig` `moduleResolution: "bundler"` 解决）。
- 文件命名：`<域>.<层>.ts`，例 `courses.handler.ts`、`courses.service.ts`、`courses.repository.ts`。

### 3.2 分层（强制）

```
HTTP → middlewares → routes(Hono sub-app) → handlers → services → repositories → supabase-js
                                                            ↘ adapters（外部服务/AI）
```

- **handler**：解析/校验 `c.req`（`@hono/zod-validator`）、调用 service、`c.json` 返回。**禁止包含业务逻辑**。
- **service**：业务逻辑核心；纯函数优先；事务边界在此层（通过 RPC 函数完成跨表事务）。
- **repository**：唯一直接持有 `supabase-js` service-role client 的层；返回领域对象，不返回原始行。
- **adapter**：唯一调用第三方/AI 的层；缺 Key 自动 mock。
- **跨层调用**：上层只可依赖下层，反向依赖一律拒绝。

### 3.3 Hono 模式

- 每个业务域一个 `Hono()` sub-app，在 `app.ts` 中 `app.route('/courses', coursesRouter)` 装配。
- 中间件优先：`requestId → logger → cors → i18n → auth → rateLimit → handler → errorHandler`。
- `errorHandler` 通过 `app.onError(...)` 注册；handler 不写 try/catch。

### 3.4 错误处理

- 业务错误使用 `errors/AppError.ts`：`new AppError({ code, message, http, details })`。
- 未知异常：log error + 返回 `code=50000`，**不向客户端泄漏栈/SQL**。
- 第三方/AI 错误：在 adapter 内归一为 `AppError(50200/50300/50400)`。

### 3.5 校验

- 入口 Zod 校验：`c.req.valid('json' | 'query' | 'param' | 'header')`；失败统一 `40001`。
- 出口可选 Zod 校验（dev 强开，prod 关闭以省 CPU）。
- 环境变量启动校验：`src/env.ts` 用 Zod；缺失关键变量直接退出。

### 3.6 日志（pino）

- 全局结构化 JSON。每条日志必须包含：`request_id`、`user_id?`、`route?`、`latency_ms?`。
- **禁止打印**：明文密码、Token、API Key、身份证号、手机号（脱敏 `138****0000`）、完整 prompt 内容（仅打 hash）。
- 级别：
  - `trace/debug`：仅 dev；
  - `info`：业务关键事件（订单创建、支付成功）；
  - `warn`：可恢复异常（限流命中、降级触发）；
  - `error`：未预期错误，必须含堆栈。

### 3.7 异步与并发

- 一律 `async/await`；禁止裸 Promise.then 链 > 1 级。
- 并发用 `Promise.all`（独立任务）/ `p-limit`（限流）。
- 长任务（> 500ms 或外部 IO）必须挂 BullMQ。

### 3.8 数据库使用

- 全部走 `packages/supabase-client`；**禁止**在业务代码 `createClient(...)`。
- 跨表写操作必须封装为 **Postgres RPC 函数**（在事务里完成），不要在应用层串多个 `insert`。
- 列表查询必须分页（默认 `page_size=20`，上限 100）。
- 禁止 `select('*')`；显式列出列。
- jsonb 写入遵守 [03-数据库规范.md §九](./03-数据库规范.md)。

## 四、Edge Functions（Deno + TypeScript）

- 入口文件：`supabase/functions/<name>/index.ts`，`Deno.serve(...)`。
- 共享代码（错误码、Zod schema、CORS）放 `supabase/functions/_shared/`，由 `scripts/sync-shared-to-edge.ts` 从 `packages/shared-schemas` 同步（Edge 不能直接 import workspace 包）。
- 鉴权与第三方回调以外的业务**禁止**写到 Edge Functions（参考 [01-技术栈.md §四](./01-技术栈.md)）。
- 同样使用 Zod 校验 + AppError 同构错误响应（与 Hono 端一致）。

## 五、测试

| 层级 | 工具 | 要求 |
|------|------|------|
| 前端单测 | Vitest + RTL | 关键交互、纯函数；快照仅用于稳定 UI |
| 前端集成 | MSW + Vitest | 真实组件 + mock 接口 |
| 后端单测 | Vitest | service / utils 层覆盖 ≥ 70% |
| 后端接口 | Vitest + Hono `app.request()` | handler 走真实路由 + 容器内 Supabase |
| Edge Functions | `deno test` | 在容器内执行 |
| 端到端（可选）| Playwright（容器内）| 关键链路：注册→购课→学课→支付（mock） |
| RLS 测试 | `supabase/tests/rls/` | 见 [03-数据库规范.md §七](./03-数据库规范.md) |

- 所有测试在 Docker 内执行：`docker compose run --rm api-app pnpm test`。
- 不允许测试连真实第三方服务，必须 mock。

## 六、Git 与提交

- 分支：`main`（受保护，仅合并）+ feature 分支 `feat/<area>-<short>`。
- 提交规范：Conventional Commits，类型限 `feat / fix / docs / style / refactor / perf / test / chore / build / ci`。
- PR 模板：链接 PRD、变更摘要、测试说明、自查清单。
- 严禁 `--no-verify`，严禁 `git push --force` 到共享分支。

## 七、安全编码（OWASP Top 10 速查）

1. SQL 注入：必须用 supabase-js 参数化或 RPC，**禁止字符串拼接**；如确需动态 SQL 走 `supabase.rpc('fn_x', { ... })` 在 SQL 内安全拼接。
2. XSS：React 默认转义；`dangerouslySetInnerHTML` 必须走 DOMPurify。
3. CSRF：纯 API + Bearer Token，无 cookie 鉴权 → 不需 CSRF Token；管理后台同。
4. 鉴权/越权：每个 handler 显式声明权限中间件；Service 层再做资源归属校验；前端禁用 service-role key。
5. 敏感信息：不入日志、不入响应；存库前加密（详见 G3 规范）。
6. 依赖：`pnpm audit` 接入 CI，高危漏洞 24h 内修复。
7. SSRF：所有外发 URL 经 allowlist 校验。
8. 文件上传：MIME 白名单 + 服务端 sniff + 大小限制；图片走 sharp 重写防止 SVG 注入。
9. 反序列化：禁止 `eval` / `new Function` 处理外部数据。
10. 速率限制：见 [04-API接口规范.md §七](./04-API接口规范.md)。
