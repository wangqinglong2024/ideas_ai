# AI 任务执行工作流 (Task Execution Workflow)

> **版本**: v1.0 | **最后更新**: 2026-04-17
>
> **本文件是 AI 执行开发任务的操作手册**。
> 覆盖：上下文工程 → 任务拆解 → Story File → 执行协议 → 增量验证 → 质量门禁。
> **核心理念**：AI 的输出质量 = f(上下文质量)。上下文不够就不动手，动手就一次做对。

---

## 一、上下文工程（Context Engineering）

> **"给 AI 喂垃圾上下文，就会得到垃圾代码。"**
> 开始任何任务前，必须先构建完整的上下文环境。这是 AI 开发质量的第一决定因素。

### 1. 上下文加载协议（每个任务启动前强制执行）

```
┌─────────────────────────────────────────────────────────┐
│               上下文加载检查清单 (Context Checklist)       │
│                                                         │
│  □ 1. 规范上下文：已读 index.md + 本任务涉及的规范文件     │
│  □ 2. 需求上下文：已读 PRD / 用户描述，理解无偏差          │
│  □ 3. 代码上下文：已搜索项目内相关文件（不少于 3 个关键词）  │
│  □ 4. 数据上下文：已查明涉及的数据库表结构和字段            │
│  □ 5. 依赖上下文：已确认上游任务完成状态和接口约定          │
│  □ 6. 环境上下文：已确认目标环境（dev/staging/prod）配置    │
│                                                         │
│  ⚠️ 任何一项为 □ 未勾选 → 禁止开始编码                    │
└─────────────────────────────────────────────────────────┘
```

### 2. 上下文依赖图

不同类型的任务需要加载不同的上下文：

| 任务类型 | 必须加载的上下文 |
|---------|---------------|
| 数据库变更 | env.md + coding-standards.md §四 + 当前 Schema（MCP 查询） |
| 后端 API | coding-standards.md §三 + api-design.md + 相关 Service/Repo 代码 |
| 前端页面 | ui-design.md + rules.md §一 + 相关组件代码 + 后端 API 文档 |
| 全栈功能 | 上述全部 + project-structure.md |
| Bug 修复 | Bug 报告 + 相关源码 + qa-testing.md |
| 性能优化 | 当前代码 + rules.md §三（高并发指引）+ 慢查询日志 |

### 3. 先查后建（Look Before You Leap）

> 写任何新代码前，必须完成以下检索。跳过此步等同于"闭眼写代码"。

```
检索顺序（按优先级）：
1. Supabase 原生能力？ → Auth / Storage / Realtime / Edge Functions
2. 框架内置能力？     → React Router / Express 中间件 / Zod
3. 项目现有代码？     → 搜索 utils/ hooks/ services/ components/
4. 成熟第三方库？     → GitHub Stars > 1000 + 最近 3 月有更新

搜索关键词策略（至少用 3 个不同角度的关键词）：
- 功能名词：如 "pagination", "wallet", "upload"
- 技术动词：如 "validate", "encrypt", "cache"
- 文件约定：如 "service.ts", "hook", "middleware"
```

## 二、任务拆解方法论

### 1. 拆解原则

| 原则 | 说明 | 违反后果 |
|------|------|---------|
| **原子性** | 每个任务 = 一个可独立验证的最小功能单元 | 无法增量验证 |
| **依赖排序** | 严格按依赖关系排列：DB → 后端 → 前端 → 集成 | 编译失败、类型缺失 |
| **可验证** | 每个任务必须有 GIVEN-WHEN-THEN 验收标准 | 无法判断是否完成 |
| **范围清晰** | 明确列出涉及的文件和不涉及的文件 | 范围蔓延 |
| **单一职责** | 一个任务只做一件事，不要"顺手"做其他的 | 回退困难 |

### 2. 标准依赖序列

```
T-1: 数据库 Schema（建表 / Migration）
  ↓
T-2: 后端 Zod Schema + TypeScript 类型定义
  ↓
T-3: 后端 Repository 层（数据访问）
  ↓
T-4: 后端 Service 层（业务逻辑）
  ↓
T-5: 后端 Router 层（API 端点）
  ↓
T-6: 前端类型同步（supabase gen types）
  ↓
T-7: 前端 API Service 层
  ↓
T-8: 前端 Hook 层
  ↓
T-9: 前端组件 + 页面
  ↓
T-10: 集成联调 + Docker 构建验证
```

### 3. 任务卡片模板

```markdown
### T-{模块}.{序号}: {任务标题}

**复杂度**: 简单(S) / 中等(M) / 复杂(L)
**预估文件数**: {N} 个

**范围（做什么）**:
- {具体要做的事情 1}
- {具体要做的事情 2}

**边界（不做什么）**:
- {明确排除的事情}

**涉及文件**:
- 新建: `path/to/new-file.ts`
- 修改: `path/to/existing-file.ts`
- 不动: `path/to/unrelated-file.ts`

**依赖**:
- 前置: T-{x}.{y} （必须先完成）
- 后续: T-{x}.{z} （依赖本任务）

**验收标准（GIVEN-WHEN-THEN）**:
1. GIVEN {前置条件} WHEN {操作} THEN {预期结果}
2. GIVEN {异常条件} WHEN {操作} THEN {错误处理}

**验证命令**:
```bash
# 编译验证
docker compose up -d --build
# 功能验证
curl http://localhost:{port}/api/v1/{endpoint}
# 或 Browser MCP 验证步骤
```

**自检重点**: [ ] 安全 [ ] 性能 [ ] 类型同步 [ ] RLS
```

## 三、Story File（复杂任务必备）

> 当任务复杂度为 L 或涉及 > 5 个文件时，必须先创建 Story File。
> Story File = 给 AI 的 "任务说明书 + 导航地图"，集中一切上下文。

### 1. Story File 存放位置

```
tasks/
├── list/
│   ├── story-{YYYYMMDD}-{简短标题}.md
│   └── ...
```

### 2. Story File 模板

```markdown
# Story: {功能标题}

> 创建日期: YYYY-MM-DD
> 状态: 📋 计划中 / 🚧 开发中 / ✅ 已完成 / 🚫 阻塞

## 需求摘要
（用 3-5 句话概括这个功能的目标和范围）

## 相关上下文
- PRD: `product/xxx.md` §{章节}
- 设计稿: Stitch 项目 {名称} / 屏幕 {名称}
- 现有代码: `src/features/{module}/` （已有 / 全新模块）
- 数据库: 表 `{table_name}` （已有 / 需新建）
- 关联任务: story-{xxx}

## 技术方案
### 数据库设计
（表结构、字段、索引、RLS 策略）

### API 设计
（端点、请求/响应格式、鉴权级别）

### 前端架构
（组件树、状态管理方案、路由）

## 任务拆解
（使用上面的任务卡片模板，按依赖序排列）

### T-1: ...
### T-2: ...

## 风险与决策
| 风险 | 应对方案 | 决策状态 |
|------|---------|---------|
| {风险描述} | {应对方案} | 已确认 / 待确认 |

## 完成检查清单
- [ ] 所有任务通过验收标准
- [ ] Docker 构建零错误
- [ ] 代码审查自检清单通过（coding-standards.md）
- [ ] 三端类型同步（DB → 后端 → 前端）
- [ ] 无残留 console.log / TODO
```

## 四、执行协议（Execution Protocol）

### 1. 两阶段法则：先计划，再执行

```
⚠️ 铁律：计划和执行必须严格分离，禁止边想边写。

Phase 1 — 计划（Plan）:
  ✅ 分析需求、搜索代码、设计方案、拆解任务
  ✅ 创建 Story File（复杂任务）
  ✅ 获得用户确认
  ❌ 禁止写任何业务代码

Phase 2 — 执行（Execute）:
  ✅ 按计划逐任务实现
  ✅ 每个任务完成后立即验证
  ❌ 禁止修改计划范围（需要调整时先暂停讨论）
```

### 2. 逐任务执行循环

```
对每个任务 T-{n}:

┌─ 1. 声明开始 ─────────────────────────────────┐
│  "开始执行 T-{n}: {任务标题}"                     │
│  确认上下文加载完整（对照上下文检查清单）            │
└───────────────────────────────────────────────┘
    ↓
┌─ 2. 编码实现 ─────────────────────────────────┐
│  按 coding-standards.md 规范编写代码              │
│  增量验证：每写完一个函数/组件立即编译检查           │
│  遇到不确定的 → 立即停下查证或提问                 │
└───────────────────────────────────────────────┘
    ↓
┌─ 3. 自检验证 ─────────────────────────────────┐
│  □ TypeScript 编译无错误                         │
│  □ Docker 构建成功（docker compose up -d --build）│
│  □ 验收标准逐条通过（GIVEN-WHEN-THEN）            │
│  □ 代码审查自检清单（coding-standards.md §五.4）   │
│  □ 无残留调试代码                                │
└───────────────────────────────────────────────┘
    ↓
┌─ 4. 声明完成 ─────────────────────────────────┐
│  "T-{n} 完成 ✅"                                │
│  简要列出新增/修改的文件                           │
│  标注下一个任务                                   │
└───────────────────────────────────────────────┘
```

### 3. 增量验证策略

> **"不要等全部写完再测，写一步验一步。"**

| 编码阶段 | 验证动作 | 失败时行为 |
|---------|---------|----------|
| 写完数据库 Migration | MCP 执行 SQL，确认表结构正确 | 立即修复 SQL |
| 写完后端 Repository | TypeScript 编译检查 | 修复类型错误 |
| 写完后端 Service | TypeScript 编译 + 单元逻辑推演 | 修复逻辑错误 |
| 写完后端 Router | `curl` 测试 API 端点 | 检查路由注册和参数绑定 |
| 写完前端组件 | `npm run build` 编译检查 | 修复 JSX/TS 错误 |
| 写完前端页面 | Docker 构建 + Browser MCP 截图 | 修复渲染和交互 |
| 集成联调 | 全链路 E2E 验证 | 按链路定位问题环节 |

### 4. 错误处理协议

```
遇到错误时的标准处理流程：

1. 读取完整的错误信息（不要只看第一行）
2. 定位错误源头（区分编译错误 / 运行时错误 / 逻辑错误）
3. 分析根因（不要只治症状）
4. 修复并验证（修复后必须重新走验证流程）
5. 记录修复内容（在 Story File 或交付清单中注明）

⚠️ 三次失败规则：
- 同一问题尝试 3 次仍未解决 → 立即停止
- 报告当前状态为 🚫 阻塞
- 附带已尝试的 3 种方案和失败原因
- 等待用户指导后再继续
```

## 五、质量门禁（Quality Gates）

> 每个阶段必须通过对应的门禁才能进入下一阶段。门禁不是建议，是强制。

### 1. 门禁总览

| 门禁 | 阶段 | 通过条件 | 失败动作 |
|------|------|---------|---------|
| Gate 0 | 上下文加载 | 6 项检查清单全勾 | 补充缺失上下文 |
| Gate 1 | 需求理解 | 用户确认理解无偏差 | 重新复述，再次确认 |
| Gate 2 | 技术方案 | 方案获用户确认 | 调整方案 |
| Gate 3 | 任务拆解 | 所有任务含验收标准 | 补充验收标准 |
| Gate 4 | 编码完成 | TypeScript 零编译错误 | 修复编译错误 |
| Gate 5 | Docker 构建 | `docker compose up -d --build` 成功 | 修复构建问题 |
| Gate 6 | 功能验证 | GIVEN-WHEN-THEN 逐条通过 | 修复功能缺陷 |
| Gate 7 | 自检清单 | coding-standards.md 自检全绿 | 补全缺失项 |
| Gate 8 | 健康评分 | QA 评分 ≥ 90 | 修复至达标 |

### 2. 代码审查自检清单（每个任务完成后执行）

```
功能正确性：
  □ 核心功能按需求正常工作
  □ 边界情况已处理（空值、空列表、超长输入、并发）
  □ 错误路径有合理的用户提示

代码质量：
  □ 命名清晰，符合 coding-standards.md 命名约定
  □ 无重复代码（相同逻辑出现 2 次以上必须抽取）
  □ 无残留调试代码（console.log / TODO 无归属）
  □ 类型完整（TypeScript 前后端均无 any）

安全性：
  □ 用户输入已校验（前端 Zod + 后端 Zod 双重）
  □ 有鉴权守卫（需登录的接口有 authMiddleware）
  □ RLS 策略已配置（新表建完就开启）
  □ 无敏感信息硬编码
  □ 涉及金额的接口有签名验证 + 幂等处理

性能：
  □ 数据库查询有索引支持
  □ 无 N+1 查询
  □ 大列表有分页

可维护性：
  □ 关键业务决策有中文注释（解释"为什么"）
  □ 数据库变更有 Migration 文件
  □ 三端类型同步（DB → Zod/TS → 前端类型）
```

## 六、交付格式

### 1. 任务级交付（每个任务完成后）

```markdown
### T-{n} 完成 ✅

**新增/修改文件**:
- `path/to/file.ts` — {简要说明}

**验收标准通过**:
1. ✅ GIVEN ... WHEN ... THEN ...
2. ✅ GIVEN ... WHEN ... THEN ...

**自检**: 安全 ✅ | 性能 ✅ | 类型同步 ✅
```

### 2. 功能级交付（所有任务完成后）

```markdown
## 交付清单

### 新增/修改文件
- `file-path` — 说明

### 验收方式
1. 启动命令：`docker compose up -d --build`
2. 访问地址：`http://115.159.109.23:{port}`
3. 操作步骤：...
4. 预期效果：...

### 完成状态
✅ 完成 / ⚠️ 完成但有顾虑 / 🚫 阻塞 / ❓ 需要上下文

### 遗留风险（如有）
- {风险描述} — {建议处理方式}
```

## 七、自我调控机制

### 1. 范围蔓延检测

```
每完成一个任务后，自查：
  □ 我是否修改了任务卡片"涉及文件"之外的文件？
  □ 我是否添加了任务卡片"范围"之外的功能？
  □ 累计修改文件数是否超过任务预估的 1.5 倍？

如果任何一项为"是" → 暂停，向用户报告范围变化
```

### 2. 复杂度升级信号

```
以下信号表明任务复杂度被低估，需要暂停重新评估：
  - 原计划 3 个文件，实际已涉及 8 个文件
  - 原计划不涉及数据库，但发现需要加字段
  - 需要修改另一个模块的代码才能完成本任务
  - 发现需求存在歧义或冲突

处理方式：暂停 → 报告 → 与用户讨论 → 调整计划 → 继续
```

### 3. WTF 风险指数

```
从 0% 开始累计：
  每次 revert:                    +15%
  每次修复触及 > 3 个文件:         +5%
  修复数超过 15 个后, 每个额外修复: +1%
  触及计划外文件:                   +20%

风险 > 20% → 强制暂停，向用户报告当前状况和建议
```
