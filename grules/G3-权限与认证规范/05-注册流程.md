## 一、邮箱 + 密码注册

```
1. 用户在 /auth/register 输入 { email, password, locale? }
2. 前端 zod 校验：
   - email 格式
   - password ≥ 8、含字母 + 数字
3. 前置接口：POST /api/auth/forgot-password-throttle 或 register-throttle（同 IP 1h ≤ 5 次注册）
4. 调 supabase.auth.signUp({ email, password, options: { data: { locale } } })
   - Supabase Auth 创建 auth.users 行，raw_app_meta_data = { role: 'user', provider: 'email' }（通过 GoTrue Hook 注入，见 §四）
   - 触发 handle_new_user trigger 自动 insert profiles
   - 发送验证邮件（自托管 SMTP）
5. 前端跳 /auth/verify-email-sent，提示用户去邮箱点链接
6. 用户点验证邮件 → /auth/callback?type=signup&access_token=...
   - 调 supabase.auth.exchangeCodeForSession 自动登入
   - 跳 /onboarding（首次：选择学习语言、HSK 起点等）→ 完成后 /
7. 失败：
   - email 已存在 → AUTH_EMAIL_TAKEN（"该邮箱已注册，请直接登录或找回密码"）
   - password 太弱 → AUTH_WEAK_PASSWORD
```

## 二、Google 自动注册

> "Google 登录"和"Google 注册"是同一个按钮、同一个流程：

```
首次 OAuth 回调时：
  - Supabase Auth 检测到 email 不在 auth.users → 自动 insert
  - email_confirmed_at = now()（Google 已验证邮箱）
  - raw_user_meta_data = { name, avatar_url, locale: 'zh'（默认）}
  - GoTrue Hook 注入 raw_app_meta_data = { role: 'user', provider: 'google' }
  - handle_new_user trigger 自动 insert profiles
  - 跳 /onboarding；二次以上登录跳 / 或 redirect
```

> 用户**无需在 UI 上点"注册"**；Google 流程自动完成。

## 三、超级管理员账号 seed

> **PM 决策**：超管账号与初始密码放在项目 `.env` 中，首次部署时由 seed 脚本创建；默认仅 1 个，但**技术上允许多个**；不提供「机器超管」 / 「运维重置」逆生通道 —— 如果丢密码或丢账号，**运维直接在 DB 手工插入一个新超管**（使用本节提供的 SQL 片段）。

### 3.1 .env 变量（与 G1 06-部署与环境.md 保持一致）

```bash
# system/.env.production / .env.local
SUPER_ADMIN_EMAIL=admin@zhiyu.com
SUPER_ADMIN_PASSWORD=<随机生成的强密码>            # 仅 seed 时读取一次，之后不再使用
SUPABASE_JWT_SECRET=<Supabase 实例生成>
SUPABASE_SERVICE_ROLE_KEY=<同上>
```

安全金线：
- `.env*` 均列入 `.gitignore`；仓库内仅提供 `.env.example`（空值）。
- 默认超管邮箱：`admin@zhiyu.com`（可覆盖）；默认初始密码**不**事先写入仓库，每环境单独生成。
- 本地 dev 允许使用固定试验密码 `Admin@123456`（仅 `NODE_ENV !== 'production'` 时，生产严禁）。

### 3.2 Seed 脚本

`system/supabase/seeds/001_super_admin.sql`：

```sql
-- 仅在初始化或新环境创建超管。邮箱已存在则跳过（幂等、安全重跑）。
do $$
declare
  admin_email text := current_setting('app.super_admin_email', true);
  admin_pwd   text := current_setting('app.super_admin_password', true);
  uid uuid;
begin
  if admin_email is null or admin_pwd is null then
    raise notice '跳过超管 seed：未提供 app.super_admin_email / app.super_admin_password';
    return;
  end if;

  if exists (select 1 from auth.users where email = admin_email) then
    raise notice '超管 % 已存在，跳过 seed', admin_email;
    return;
  end if;

  -- 通过 supabase_auth_admin schema 函数创建（见 supabase/migrations/000_auth_helpers.sql）
  uid := public.create_user_with_password(admin_email, admin_pwd);

  update auth.users
  set raw_app_meta_data = jsonb_build_object('role', 'super_admin', 'provider', 'email'),
      email_confirmed_at = now()
  where id = uid;
end $$;
```

### 3.3 执行方式

```bash
# 部署时通过环境变量传入
psql "$DATABASE_URL" \
  -v "ON_ERROR_STOP=1" \
  -c "SET app.super_admin_email = '$SUPER_ADMIN_EMAIL'; SET app.super_admin_password = '$SUPER_ADMIN_PASSWORD';" \
  -f system/supabase/seeds/001_super_admin.sql
```

集成到 `pnpm db:seed` 脚本（详见 G1 06）；`docker-compose up` 首次启动时也会自动运行。

### 3.4 手工创建额外超管（逆生）

当唯一超管账号密码丢失或账号损坏时，运维可直接在 DB 控制台运行：

```sql
SET app.super_admin_email = 'ops-rescue@zhiyu.com';
SET app.super_admin_password = '<临时密码>';
\i system/supabase/seeds/001_super_admin.sql
-- 登录后手动修改密码并禁用原账号
```

新超管同样需「首次登录 → 绑定 TOTP」（见 02 §1.3）。

### 3.5 超管自我保护

- 管理端「用户管理」列表中，超管自身那行：
  - 「禁用 / 启用」按钮 → 隐藏；
  - 「删除」按钮 → 隐藏；
- 后端 `DELETE /api/admin/users/:id`：
  ```ts
  if (id === c.get('user').id) return jsonError(c, 400, 'AUTH_SUPER_ADMIN_SELF_DELETE');
  ```
- `PATCH /api/admin/users/:id/disable` 同样校验：**禁止禁用自己**；允许禁用其他 `super_admin`（多超管场景，如临时逆生后需禁用原账号）。
- 本期管理端 UI **不提供「创建超管」入口**；需多超管请走 §3.4 手工路径。

## 四、GoTrue Hook：注册前风控 + 默认 role注入

> **PM 决策（Q3）**：使用 Supabase Auth 原生 `before_user_created` Hook，在账号写入前一次性完成「邮箱 / IP / 邀请码 / 风险规则 + 默认角色」检查；若部署的 GoTrue 镜像不支持 → 降级为 DB trigger，仅在账号创建后做初始化与标记（不能阻断创建）。部署前必须确认 GoTrue 版本 ≥ v2.130（支持 hook）。

### 4.1 主方案：GoTrue `before_user_created` Hook

`system/apps/api/src/routes/internal/auth-hook.ts`（Hono 路由，HMAC 验签）：

```ts
import { Hono } from 'hono';
import { timingSafeEqual } from 'node:crypto';

export const authHook = new Hono();

authHook.post('/internal/auth-hook', async (c) => {
  // 1. HMAC 签名验证
  const sig = c.req.header('x-supabase-signature') ?? '';
  const body = await c.req.text();
  if (!verifyHmac(body, sig, process.env.SUPABASE_AUTH_HOOK_SECRET!)) {
    return c.json({ error: 'invalid signature' }, 401);
  }

  const { event, user, request } = JSON.parse(body);
  if (event !== 'before_user_created') return c.json({ decision: 'continue' });

  // 2. 邮箱与 IP 风控
  if (await isEmailBlacklisted(user.email)) {
    return c.json({ decision: 'reject', message: 'email blacklisted', code: 'AUTH_EMAIL_BLOCKED' });
  }
  if (await isIpBlacklisted(request?.ip)) {
    return c.json({ decision: 'reject', message: 'ip blocked', code: 'AUTH_IP_BLOCKED' });
  }
  if (await ipRegisterCountInLastHour(request?.ip) >= 5) {
    return c.json({ decision: 'reject', message: 'rate limited', code: 'AUTH_REGISTER_RATE_LIMITED' });
  }

  // 3. 邀请码（可选）——仅当前台开启邀请制时才检
  const inviteCode = user.user_metadata?.invite_code;
  if (process.env.INVITE_ONLY === 'true') {
    if (!inviteCode || !(await consumeInviteCode(inviteCode))) {
      return c.json({ decision: 'reject', message: 'invalid invite', code: 'AUTH_INVITE_INVALID' });
    }
  }

  // 4. 默认角色与 provider 注入
  return c.json({
    decision: 'continue',
    user_metadata: user.user_metadata,
    app_metadata: {
      role: 'user',
      provider: user.app_metadata?.provider ?? 'email',
    },
  });
});
```

配置：`system/supabase/config.toml`：

```toml
[auth.hook.before_user_created]
enabled = true
uri = "https://api.zhiyu.com/internal/auth-hook"
secret = "env(SUPABASE_AUTH_HOOK_SECRET)"
```

### 4.2 降级方案：DB Trigger（仅初始化与标记，不作风控）

如 GoTrue 版本不支持，在 `handle_new_user` 中补充默认 role：

```sql
create or replace function public.handle_new_user()
returns trigger language plpgsql security definer as $$
begin
  -- 原有 profiles insert ...
  -- 补充 role 写入，避免覆盖已有值
  if new.raw_app_meta_data->>'role' is null then
    update auth.users
    set raw_app_meta_data = coalesce(raw_app_meta_data, '{}'::jsonb)
      || jsonb_build_object('role', 'user', 'provider', coalesce(raw_app_meta_data->>'provider', 'email'))
    where id = new.id;
  end if;
  return new;
end $$;
```

降级后上述「邮箱 / IP / 邀请码风控」全部上提到 Hono 的「注册节流接口 `POST /api/auth/register-throttle`」，前端在调 supabase.auth.signUp 前必须先调（会错过服务间直接调 GoTrue 的场景，但本项目不存在该场景）。

### 4.3 选型决策点

- 部署脚本中检测 GoTrue 版本：`curl $SUPABASE_AUTH_URL/health` → 解析 `version`；
- ≥ v2.130 走 4.1；不足走 4.2 并记录到部署报告。
- 选型结果写入 `system/.deploy-status.json`，其他服务可读取判断。

## 五、注册节流与防滥用

> **PM 决策（Q1）**：本期不接 Cloudflare Turnstile / hCaptcha 等第三方，仅靠后端节流与 GoTrue Hook 风控。

| 维度 | 限制 |
|------|------|
| 同 IP 注册 | 1 小时 ≤ 5 次（Hook 与 Hono 双层塑装）|
| 同 IP 邮箱验证邮件 | 1 小时 ≤ 5 次（与上面共用计数）|
| 同 email 注册尝试 | 1 小时 ≤ 3 次 |
| 邮箱黑名单（一次性邮箱 / 已知滥用域）| GoTrue Hook 拒绝创建 |
| 邀请码（可关闭）| `INVITE_ONLY=true` 时，无有效邀请码 → 拒绝 |

实现：Hono 路由 `POST /api/auth/register-throttle`；Redis key=`reg:ip:{ip}` / `reg:email:{email}`，TTL 3600s。前端在调 supabase.auth.signUp 前先调，后端 Hook 再作二次拦截（防服务间绕过）。
